SECURITY ARCHITECTURE
================================================================================

MULTI-TENANCY ISOLATION:
├─ Row-Level Security (RLS) - Database-enforced isolation
│  └─ Every query automatically filters to current org_id
├─ RLS Policies on 14 tables
│  └─ Zero cross-tenant data leakage possible
└─ No application code needed for isolation

ENCRYPTION:
├─ At Rest:
│  ├─ Sensitive columns encrypted with AES-256
│  ├─ Password hashes with bcrypt
│  ├─ MFA secrets encrypted in database
│  └─ API keys encrypted column-level
├─ In Transit:
│  ├─ TLS 1.3 for all connections
│  ├─ Client certificates for extra hardening
│  └─ Connection pooling with encrypted channels
└─ Key Management:
   ├─ KMS master keys (AWS)
   ├─ Data encryption keys rotated monthly
   └─ Passwords rotated every 90 days

AUDIT & COMPLIANCE:
├─ Immutable Audit Log:
│  ├─ Every mutation logged automatically (trigger-based)
│  ├─ Before/after state snapshots (JSONB)
│  ├─ SHA-256 tamper-evident hash chain
│  └─ Monthly partitions for performance
├─ Audit Trail Covers:
│  ├─ CREATE, UPDATE, DELETE operations
│  ├─ User identity & timestamp
│  ├─ IP address & user agent
│  └─ Complete state before & after
└─ Tamper Detection:
   └─ Hash chain verification proves no data modified

ROLE-BASED ACCESS CONTROL:
├─ 8+ Role Types:
│  ├─ Office Manager (org admin)
│  ├─ CFO (full access)
│  ├─ Finance Controller (GL + approvals)
│  ├─ Accountant (GL management)
│  ├─ VAT Specialist (VAT filing)
│  ├─ Tax Specialist (tax planning)
│  ├─ Agent Manager (AI agent control)
│  └─ Support Team (help desk)
├─ Permission System:
│  ├─ Granular permissions (read:org, write:gl, file:vat_return, etc.)
│  ├─ Approval workflows (can_approve flag)
│  ├─ Filing permissions (can_file_returns flag)
│  └─ Temporary role assignments (expires_at)
└─ Enforcement:
   ├─ Database layer: RLS policies
   ├─ Application layer: Middleware checks
   └─ API layer: Controller decorators

PERFORMANCE CHARACTERISTICS
================================================================================

QUERY LATENCIES (with proper indexes):
├─ Single record by ID ......... 1-5ms    (primary key lookup)
├─ List by org (100 records) ... 10-30ms  (index scan)
├─ Aggregation (SUM) ........... 50-100ms (full partition scan)
├─ Cross-table JOIN (3 tables) . 30-80ms  (query planner optimized)
└─ Audit trail search .......... 100-200ms (partitioned scan)

CONCURRENCY:
├─ Connection pooling: 25 per app × N instances
├─ PgBouncer transaction-mode pool: 1000+ concurrent users
├─ Max connections: 200 on primary
└─ Recommended: 3 read replicas for analytics

SCALABILITY:
├─ 0-1k orgs:     Single primary + 2 replicas
├─ 1k-10k orgs:   Vertical scaling (64GB RAM)
├─ 10k-100k orgs: Sharding by org_id hash
└─ 100k+ orgs:    Multi-region shards

EXPECTED THROUGHPUT:
├─ At scale: 10,000+ queries per second
├─ With 3 read replicas: 30,000+ QPS possible
└─ Audit log: Partitioned by month for performance
